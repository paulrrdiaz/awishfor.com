## Context

Root layout (`src/app/layout.tsx`) currently sets `metadata.icons` to a single entry pointing at `public/favicon.ico`, which is the unmodified create-t3-app placeholder. `public/site.webmanifest` does not exist yet. A generated icon set (favicon.ico, favicon.svg, favicon-96x96.png, apple-touch-icon.png, web-app-manifest-192x192.png, web-app-manifest-512x512.png, site.webmanifest) already exists at `/Users/paulrrdiaz/Downloads/favicon`, produced by a standard favicon generator, with placeholder `name`/`short_name` values in its manifest.

## Goals / Non-Goals

**Goals:**
- Serve the branded icon set across browser tab, iOS home screen, and Android/PWA install surfaces.
- Wire icons + manifest into Next.js `metadata` so `<link>` tags are emitted automatically, no manual `<head>` markup.
- Fix manifest placeholder strings (`MyWebSite`/`MySite`) to real product name.

**Non-Goals:**
- No PWA feature work (service worker, offline, install prompts) — manifest is added only for icon/identity metadata.
- No design/brand exploration — icon assets are taken as-is from the provided generator output; regenerating or redesigning the mark is out of scope.
- No favicon.ico removal from Next.js file-convention path (`src/app/favicon.ico`) — this repo already serves icons via `public/` + explicit `metadata.icons`, so that pattern is kept for consistency rather than switching to Next's file-based icon convention.

## Decisions

- **Copy assets into `public/` as-is, overwrite `public/favicon.ico`.** The existing repo already serves `favicon.ico` from `public/` and references it explicitly via `metadata.icons`, rather than using Next.js's `src/app/favicon.ico` auto-convention. Keeping the same pattern (`public/` + explicit metadata) avoids mixing two icon-resolution mechanisms.
- **Declare icons via the `metadata.icons` object form** (`icon`, `shortcut`, `apple` arrays) instead of a flat array, so favicon.ico, favicon.svg, and favicon-96x96.png can all be listed with correct `type`/`sizes`, and the Apple touch icon gets its own `rel="apple-touch-icon"` link. Alternative considered: keep the single-array form used today — rejected because it can't express distinct `rel`/`sizes` per icon.
- **Add `manifest: "/site.webmanifest"` to `metadata`** so Next.js emits `<link rel="manifest">` automatically, instead of hand-writing the tag in `layout.tsx`.
- **Edit `site.webmanifest` `name`/`short_name` to "A Wish For"** before copying it in; leave `theme_color`/`background_color`/`display`/icon `purpose` as generated, since no brand color decision is part of this change.
- **Also fix `metadata.title`/`description` placeholders** (`"Create T3 App"` / `"Generated by create-t3-app"`) in the same edit to `layout.tsx`, since they sit in the same object as the icons change and shipping a branded favicon next to an unbranded page title would be an inconsistent half-fix. Scoped to the root metadata literal only — no per-page metadata overrides.

## Risks / Trade-offs

- [Overwriting `public/favicon.ico` clobbers the current placeholder without a diff-visible rename] → acceptable; git history retains the old file, and the new one is the intended replacement.
- [Manifest `theme_color`/`background_color` still generator defaults (`#ffffff`), not verified against actual brand palette] → out of scope per Non-Goals; flagged here so a future brand pass can revisit.
- [Editing `metadata.title`/`description` slightly broadens the change beyond "just the favicon"] → mitigated by keeping the copy change minimal (real product name/description only, no new metadata fields like Open Graph).
